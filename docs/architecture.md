# アーキテクチャと技術選定の根拠

最終更新: 2026-07-02

このドキュメントは、なぜこの構成に決めたかの意思決定記録（ADR 的なメモ）を兼ねる。

---

## 1. 設計の中核思想

### 1.1 事前計算 → 静的配信（v1）
流域導出も InVEST も重い処理。最初からリアルタイム地理演算サーバーで作ると運用と費用が破綻する。
v1 は **対象地域を決めたらバッチで一括計算し、結果をタイル＋統計に焼き込んで静的配信**する。
タイルサーバーも DB も v1 では持たない。

動的化（任意地域をその場で／自前データ上書き）が要る **v1.1** で初めてサーバーを足す。

### 1.2 静的タイルと動的サービスの分離
「静的タイルをどこに置くか」と「動的サービスをどこで走らせるか」を分離する。
静的タイルは **Cloudflare R2（エグレス無料）に固定**。これにより、
将来動的ホストを何にしても**タイル配信費はゼロのまま**で、ホスト選定の失敗コストが下がる。

## 2. コンポーネント構成

```
┌─────────────────────────────────────────────────────────┐
│ ブラウザ                                                  │
│   Next.js (Vercel) + MapLibre GL JS + pmtiles protocol   │
└───────────────┬─────────────────────────┬───────────────┘
                │ range requests           │ (v1.1) API
                ▼                          ▼
     ┌────────────────────┐     ┌──────────────────────────┐
     │ Cloudflare R2 + CDN │     │ Fly.io（v1.1〜・仮置き）  │
     │  *.pmtiles / *.json │     │ FastAPI + TiTiler + PostGIS│
     └────────────────────┘     └──────────────────────────┘
                ▲
                │ アップロード（成果物）
     ┌────────────────────┐
     │ 地理処理パイプライン │  ← ローカル or ジョブ実行（常時稼働なし）
     │  Python バッチ      │
     └────────────────────┘
```

## 3. レイヤ別の採用と理由

### 3.1 フロント: Next.js + MapLibre GL JS
- **Next.js**: 将来の軽量 API・レポート画面を同居させやすい。ユーザーが選定。
- **MapLibre GL JS**: OSS。ベクタタイル・hillshade・terrain が揃い、YAMAP 風の地形表現が出せる。
  Mapbox GL JS は有償ライセンスのため回避。
- **PMTiles**: タイル群を単一ファイルに固めた形式。HTTP Range で必要部分だけ読むため
  **タイルサーバー不要**。`pmtiles` プロトコルプラグインを MapLibre に登録して直読み。
- deck.gl は v1 では使わない（コロプレスは MapLibre の fill 式で足りる）。3D 押し出し等が
  欲しくなったら追加。

### 3.2 地理処理: Python バッチ
- **rasterio / geopandas / shapely**: ラスタ・ベクタの標準。
- **pysheds**: DEM からの流域導出。pip 一発・純 Python で導入が軽い。ユーザーが選定。
  都市部の平坦地形で集水域がうまく切れない場合は WhiteboxTools の窪地 breaching に差し替える。
- **natcap.invest**: 炭素・水収量などの自然資本モデル。**査読済み＝数値の出典を示せる**。
  自前ロジックで金額や係数を捏造しないための土台。
- **土地被覆は ESA WorldCover 10m（既製）を利用**。自前の Sentinel-2 ML 分類は v1 では作らない。
- **タイル化**: ベクタは tippecanoe、陰影は gdaldem → いずれも PMTiles 化。

### 3.3 静的配信: Cloudflare R2 + CDN
- **エグレス無料**が決め手。地図タイルは読み出しが多く、従量エグレスのプラットフォームに
  置くと利用増で費用が跳ねる。R2 固定でこのリスクを土俵から外す。
- 10GB 保管無料。v1 のタイルは数十〜数百 MB 規模で実質無料。

### 3.4 フロント配信: Vercel Hobby
- Next.js 純正で最も滑らか。個人開発フェーズは無料。
- 注意: **Hobby は非商用ライセンス**。収益化フェーズで Vercel Pro か他所へ移す。
  Cloudflare に寄せたい場合は Cloudflare Pages（`next-on-pages`）も選択肢。

### 3.5 動的サービス（v1.1〜）: Fly.io（仮置き）
- **Fly.io** に FastAPI + TiTiler（COG からラスタタイル）+ PostGIS を載せる想定。
  - **scale-to-zero** でアイドル時の課金が止まる（個人開発に有利）。
  - エグレス寛容、非商用制約なし、コンテナをそのままデプロイ。
- 最安を極めるなら Hetzner VPS（€4/月〜・転送量潤沢・要 Docker 運用）が対抗。
- v1.1 着手時に最終確定する。**静的タイルは R2 のまま据え置き**、動的ホストは
  「その場生成タイル＋API」だけを捌くため転送量は小さい。

## 4. 却下した選択肢

| 選択肢 | 却下理由 |
|---|---|
| Mapbox GL JS | 有償ライセンス。MapLibre で代替可。 |
| 最初から動的タイルサーバー（Martin/pg_tileserv）+ PostGIS | v1 は固定地域・月次更新で不要。常時稼働＝無駄なコスト。 |
| Railway（動的ホスト） | DX は良いがエグレス従量・scale-to-zero なし。個人開発のコスト最小化では Fly.io に劣る。 |
| 自前 Sentinel-2 土地被覆 ML 分類（v1） | WorldCover 既製で足りる。ML は精度検証コストが高く v1 では過剰。 |
| 自然資本の金額換算（v1） | 換算係数の妥当性・受容性検証が必要。生物物理量に留めて後回し。 |

## 5. コスト見通し（個人開発フェーズ）

| 項目 | v1 | v1.1〜 |
|---|---|---|
| フロント（Vercel Hobby） | 0 円 | Pro 移行時に課金 |
| 静的タイル（R2） | ほぼ 0 円（エグレス無料） | 同左 |
| パイプライン | 0 円（ローカル実行） | ジョブ実行分のみ |
| 動的サービス（Fly.io） | — | scale-to-zero で低額（使用時のみ） |

v1 は実質無料で立ち上げ、課金は「本当に常時稼働が必要になった部分」にだけ発生する。
